# Foresight — Data Pipeline (M1)

Stripe events flow from the ingestion service (M0) through a medallion
lakehouse and out as per-tenant KPI time series.

```
Kafka / Event Hubs  (stripe.events.<tenant_id>)
        │
        ▼  Spark Structured Streaming  (streaming/)
   Iceberg BRONZE   lakehouse.bronze_stripe_events   ← raw, immutable, append-only
        │
        ▼  dbt  (dbt/foresight/)
   SILVER  stg_stripe_events        ← typed, deduped, JSON parsed
        │
        ▼
   intermediate  int_metric_events  ← KPI contributions per event
        │
        ▼
   GOLD  fct_kpi_timeseries         ← per-tenant, per-day MRR / churn / refund /
                                       payment-failure / conversion
```

## Design decision: streaming lands bronze, dbt builds silver + gold

The build plan lists silver/gold under both Spark and dbt. We split the
responsibility to avoid duplicated logic:

- **Spark Structured Streaming** does the real-time, stateful-I/O part: durable,
  exactly-once landing of raw events into **bronze**. No business logic.
- **dbt** owns **silver → gold** as version-controlled, tested SQL. Every KPI
  definition lives in exactly one model.

This is the standard lakehouse division of labor (streaming ingest + batch
transform) and keeps the KPI logic reviewable and unit-tested.

## `streaming/` — bronze ingest

PySpark Structured Streaming job (`foresight_streaming/bronze_ingest.py`). The
projection logic (`transform.to_bronze`) is a pure DataFrame→DataFrame function
and is unit-tested on a local SparkSession — no Kafka or Iceberg required:

```bash
cd pipelines/streaming
pip install -e ".[dev]"
pytest -q          # requires a JVM (Java 17)
```

The full job runs on Databricks against Event Hubs + Iceberg. Bronze is
partitioned by `(tenant_id, event_date)`; raw Stripe objects are stored verbatim
as JSON in `payload` to preserve type fidelity.

## `dbt/foresight/` — silver + gold

Runs on **DuckDB** locally/CI (fast, no JVM) and on **Databricks** in
production. Engine-specific SQL (JSON extraction, epoch→timestamp) is abstracted
behind adapter-dispatched macros (`macros/`), so the model SQL is identical
across engines.

```bash
cd pipelines/dbt/foresight
pip install "dbt-duckdb>=1.8,<2.0"
dbt seed --profiles-dir .     # loads the bronze fixture
dbt build --profiles-dir .    # models + tests
```

The seed (`seeds/bronze_stripe_events.csv`, generated by
`_generate_bronze_seed.py`) is a hand-crafted three-day, two-tenant Stripe event
stream. `tests/assert_alpha_known_kpis.sql` pins the exact expected KPI values,
so any drift in the MRR/churn/refund/conversion logic fails CI.
